# DR: imgcat によるデュアルバー描画

## ステータス: 検討中（PoC 未実施）

## 背景

### 現行方式: quadrant 文字 + ANSI 2色

デュアルバーは 1 セルに上段（utilization）と下段（elapsed time）の 2 本のバーを描画する。
現行は Unicode の quadrant 文字（`▀▄▘▖▌▛▙█` 等）で 4 象限を表現するが、ターミナルの制約により **1 セルあたり fg + bg の 2 色しか使えない**。

### 2 色制限による妥協

4 象限（TL, TR, BL, BR）各々が U（使用）/ A（空き）の 2 状態 → 2^4 = 16 通り。
ただし上段 AA/UA/UU × 下段 AA/UA/UU = 9 通りに正規化される。

このうち 3 状態で色の妥協が必要:

| 状態 | 理想 | 現行の妥協 |
|------|------|------------|
| UAUA | TL=TU, TR=TA, BL=BU, BR=BA (4色) | top 優先: `▘` fg=TU bg=BA → 下段境界が不正確 |
| UAUU | TL=TU, TR=TA, BL=BU, BR=BU (3色) | bottom 優先: `▄` fg=BU bg=TA → 上段境界が不正確 |
| UUUA | TL=TU, TR=TU, BL=BU, BR=BA (3色) | top 優先: `▀` fg=TU bg=BU → 下段境界が不正確 |

実用上は「バーが右から左に進む過程で一瞬ちらつく」程度だが、正確な描画には 4 色が必要。

## 提案: iTerm2 Inline Image Protocol

### 概要

iTerm2 の Inline Image Protocol (`\x1b]1337;File=...`) を使い、各セルを小画像として描画する。
画像なら 4 象限を正確に 4 色で塗り分けられる。

### 画像生成

1 セル = 2x2 ピクセルの PNG 画像（最小単位）。

#### パターンの組み合わせ

| 要素 | 種類 | 内訳 |
|------|------|------|
| ブロックパターン | 16 | TL,TR,BL,BR 各 U/A の全組み合わせ |
| テキスト文字 | 14 | `0-9`, `%`, `h`, `d`, ブランク |
| TU カラー | 3 | 赤(196) / 黄(220) / 緑(40) |

**合計: 16 x 14 x 3 = 672 枚**

#### テキスト色

背景の明度で黒/白を自動選択（2 択）。
テキスト色は画像生成時に背景色から決定するため、組み合わせ数には影響しない。

#### カラー定義（現行と同一）

| エイリアス | 用途 | 256色コード | 概算 RGB |
|-----------|------|------------|----------|
| TU (赤) | 上バー使用部 (util >= 80%) | 196 | #ff0000 |
| TU (黄) | 上バー使用部 (util >= 50%) | 220 | #ffd700 |
| TU (緑) | 上バー使用部 (util < 50%) | 40 | #00d700 |
| TA | 上バー空き | 240 | #585858 |
| BU | 下バー使用部 | 39 | #00afff |
| BA | 下バー空き | 24 | #005f87 |

### 画像の生成と配布

#### 案 A: ランタイム生成 + キャッシュ

初回実行時に 672 枚を生成し、base64 エンコードした辞書を JSON ファイルとして保存。
2 回目以降は辞書を import して使う。

- 利点: リポジトリサイズが小さい
- 欠点: 初回生成に依存ライブラリ（canvas / sharp 等）が必要

#### 案 B: ビルド時生成 + TS 埋め込み

ビルドスクリプトで 672 枚を生成し、base64 文字列の辞書を `.ts` ファイルに埋め込む。

- 利点: ランタイム依存なし、single file 維持
- 欠点: ファイルサイズ増加（672 枚の base64）

#### 案 C: 最小 PNG をコードで直接生成

2x2 ピクセル程度の PNG なら、ヘッダ + IDAT を手書きで組み立てられる。
zlib 圧縮は Bun の `Bun.deflateSync` で対応可能。

- 利点: 外部依存ゼロ、ランタイムで動的生成、キャッシュ不要
- 欠点: PNG バイナリ構造の実装が必要（ただし 2x2 なら単純）

### セル画像のサイズ

iTerm2 Inline Image の `width=1` / `height=1` でセル単位指定が可能。
画像のピクセル数はセル内でスケーリングされるため、2x2 でも 100x100 でも見た目は同じ。

テキスト文字を描画する場合は、フォントレンダリングが必要なためピクセル数を増やす必要がある。
推定: 1 セルあたり 8x16 ~ 16x32 ピクセル程度。

## 未検証事項（PoC で確認が必要）

### P0: Claude Code statusline が escape を通すか

Claude Code の statusline 出力パイプラインが `\x1b]1337;File=...` を透過するか不明。
フィルタリングされている場合、この方式は使えない。

**検証方法**: statusline.ts から最小の imgcat escape を出力してみる。

### P1: セル単位の画像を隙間なく並べられるか

10 セル分の画像を横に並べたとき、セル間に隙間が生じないか。
iTerm2 の inline image は通常 1 枚の大きな画像として使われるため、セル単位の並びは未検証。

**検証方法**: 色付き 1 セル画像を 10 個連続出力し、隙間の有無を目視確認。

### P2: テキスト文字の可読性

2x2 ピクセルでは文字描画は不可能。テキスト付きセルは解像度を上げる必要があるが、
セルサイズ指定 `width=1` でのフォントレンダリング品質が十分か。

**検証方法**: 8x16 / 16x32 でテキスト入り画像を生成し可読性を確認。

### P3: パフォーマンス

672 枚の base64 文字列辞書の読み込みと、10 セル分の escape sequence 出力が
statusline の更新レート（数百 ms 以内）に収まるか。

## 代替案

### ANSI のまま改善

妥協セルの出現頻度は低い（バー進行の一瞬）ため、現行のままでも実用上問題なし。
「正確な 4 色描画」が必須要件でなければ、この改善は不要。

### Kitty Graphics Protocol（推奨）

ユーザーのターミナルは Ghostty。Ghostty は Kitty Graphics Protocol を
ネイティブ対応しており、iTerm2 の OSC 1337 より推奨される。

- APC (`\x1b_G...`) ベースのプロトコル
- PNG/RGB/RGBA の直接送信が可能
- 画像の配置・サイズ指定が柔軟（セル単位指定可能）
- Ghostty / Kitty / WezTerm 等で広くサポート

imgcat 方式を採用する場合は Kitty Graphics Protocol を第一候補とする。

## 決定事項

- ターミナル: Ghostty（Kitty Graphics Protocol 対応）
- 画像方式は一旦保留。現行 ANSI quadrant 方式で運用
